# Spec-Driven Development: From Code to Contract in the Age of AI Coding Assistants

> [!info] Source metadata
> **Author:** Deepak Babu Piskala · **Venue:** arXiv:2602.00180 (also OpenReview id bw5mNj75h9) · **URL:** https://arxiv.org/abs/2602.00180

## Key takeaways

- Core thesis: AI coding assistants make it possible to **invert the traditional workflow** — "spec-driven development (SDD) inverts the traditional workflow by treating specifications as the source of truth." Code becomes a generated or verified *secondary* artifact.
- SDD is framed as a move "from code to contract": the durable, authoritative artifact is the specification (the contract), not the code.
- **Three-tier rigor taxonomy:** `spec-first` → `spec-anchored` → `spec-as-source`. Each tier applies to different contexts with increasing rigor; the spec's authority over code increases up the ladder.
- Modern tools (GitHub Spec Kit, BDD frameworks) operationalize the spec-first philosophy.
- Provides a **decision framework** for when SDD adds value vs. when simpler approaches suffice — utility varies by domain (API, enterprise, embedded) and project characteristics.

## Notable claims (with location)

- "Specifications as the source of truth" (abstract) — code derives from or is validated against the spec.
- Case studies span API development, enterprise systems, and embedded software domains.
- SDD's value is contingent, not universal — the paper explicitly resists treating it as a default for all work.
